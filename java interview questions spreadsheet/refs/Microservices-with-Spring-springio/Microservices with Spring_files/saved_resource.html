                                                                        Paul Chapman Sam 5 years ago

                                                                        Good question. I'm not actually sure you need Eureka. If each microservice is fed by its own queue(s), all you need to do is send a request to the right queue, let the microservice read and process the message and wait for a response (if there is one).                                                                                 Paul Chapman RuRu 4 years ago

                                                                                Again this would make a good subject for a second blog, but essentially, each microservice would read from a dedicated input queue (or queues), processing messages and returning results. No service discovery is needed so long as each client process knows the name(s) of the queues(s) used by each microservice. The big difference here is that the clients can become asynchronous, the RESTful approach I describe here is synchronous.

                                                                                                                                      Paul Chapman Kishore a year ago

                                                                                                                                      Accounts are managed by two different microservices. Each can have its own view of that data to suit its needs. To reflect that they have their own Account definition. Remember one goal of microservices is that each can be developed and maintained separately to its own project schedule. If they share domain objects, which may seem an obvious thing to do, you are tying them together.

                                                                                                                                      To be fair, there is a body of opinion that putting common domain objects in separate shared project is the better way to go. Duplication or dependency. Your call.
