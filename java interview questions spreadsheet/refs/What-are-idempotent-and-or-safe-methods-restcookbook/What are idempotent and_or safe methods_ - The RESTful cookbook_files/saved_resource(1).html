                                        Mladen ericwyles • 2 years ago • edited

                                        I believe the 'result' is about the effect on the server side on the resource itself, like you said. It shouldn't have anything to do with the page/status returned. One good example of an idempotent method is a GET method which returns the current time. Obviously, the content of the page returned is always different, but the effect on the server is the same (the state of the resource is still the same - it's still the concept of the current time). Using PUT, you could update that resource to be 'current DST time' for example, instead of current time and subsequent GETs would now get something other than they used to get before. Hence, the resource state is changed.

                                              Barney Laurance Antonio • a year ago • edited

                                              The HTTP spec points out that never changing thigns on the server is unrealistic. The point of a safe method is that they shouldn't represent a *request* from the client to change things. The changes can be made if the operaters of the server want to have it programmed to change stuff, but they shouldn't assume that the operator of the client wanted that change or hold them accountable for it.

                                              Maybe a web site has a list of the most popular pages on it. It would make sense to update that list based on frequency of GET requests for each page. The user has just requested to see the page, not to have the list updated, but the latter happens anyway because its what the owners of the website want to happen.
